# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-based editor for **AI-generated static HTML presentations** (decks built from `index.html` + `style.css` + `script.js` + optional `three_scene.js`). It renders a deck in an isolated `<iframe>` and layers selection, drag/resize, text-node editing, and AI patches on top — without touching the deck's own CSS/JS, which stay loaded and authoritative.

React + Vite + TypeScript app, deployable as static files plus one serverless function. It began as a PyQt6 desktop prototype (`QWebEngineView` + `QWebChannel`); that has been fully removed and replaced — there is no Python left. `docs/web-migration-plan.md` records the design rationale.

## Commands

```bash
npm install
npm run dev            # builds the editor bundle, then Vite dev server (http://localhost:5173)
npm run build          # editor bundle + tsc + production app build -> dist/
npm run typecheck      # tsc, no emit
npm run build:editor   # rebuild ONLY the injected editor bundle -> public/editor-bundle.js
```

There is no test suite yet.

**The editor-bundle gotcha (important):** the editor runs *inside the deck iframe*, so it is built as a separate IIFE (`vite.editor.config.ts` → `public/editor-bundle.js`) and injected at runtime. **Vite HMR does not cover it.** After editing anything under `src/editor/*` you must `npm run build:editor` (or restart `npm run dev`) to see the change. The app shell under `src/app/*` hot-reloads normally.

## The core invariant — one safe patch path

Both the Inspector panel and AI Edit converge on the same editor function, `applyPatch()` (`src/editor/core.ts`). The AI is never trusted to emit HTML/CSS/JS — only a small JSON patch with a fixed key set, sanitized **server-side and again client-side** before it reaches the DOM.

The patch contract lives in **one place**, `shared/editing/` (a barrel re-exported as `@shared/editing`), imported by every consumer (this is the big structural win over the old PyQt code, where the key list was duplicated in three files). The `shared/` tree is split by domain: `shared/editing/` (the patch/action vocabulary, below) and `shared/generation/` (the deck-generation contract — see that section):

- `shared/editing/patch-keys.ts` — `PATCH_KEYS` + the `Patch` / `PatchKey` types (the single source of truth).
- `shared/editing/validator.ts` — `validatePatch()`, the sanitization gate. Drops unsafe/unknown values silently; never throws.
- `shared/editing/patch-schema.ts` — the OpenAI structured-output JSON schema, built from `PATCH_KEYS`.

The same principle now extends past single-object patches to a richer **action envelope** (`shared/editing/actions.ts`) — `patch` | `layout` | `insertBlock` — so the AI can arrange objects and add content **without ever emitting HTML**. `validateActions()` is the gate; `editorApi.applyActions()` runs a whole batch under one undo snapshot. See `docs/ai-actions-plan.md` and the "AI actions" section below.

Adding or removing an editable property is a one-line change in `shared/editing/patch-keys.ts`; the validator, schema, and `applyPatch` all key off it. Keys: `text, x, y, w, h, fontSize, color, backgroundColor, borderColor, borderWidth, borderStyle, borderRadius, fontWeight, lineHeight, letterSpacing, opacity, zIndex, filter, src, backgroundImage, animationName, animationDuration, animationDelay, animationTimingFunction, animationIterationCount, animationPlayState`.

### Animations (preset library — Path A)

The AI/Inspector can animate objects, but only by **picking from a fixed menu**, never by authoring CSS/JS — so the trust model is unchanged. The menu lives in `shared/editing/animation-presets.ts` (the curated 8: `fadeIn, fadeInUp, fadeInDown, slideInLeft, slideInRight, zoomIn, pulse, float`), which exports the preset name list (the enum), the timing-function enum, per-preset defaults, and the `@keyframes` CSS — imported by the validator (`safeAnimationName` is a strict whitelist), the JSON schema (`animationName`/`animationTimingFunction` are `enum`s), the editor, and the AI prompt. `applyAnimation` (`src/editor/animation.ts`, the animation slice of `applyPatchToElement`) writes inline `animation-*` longhands referencing namespaced `@keyframes hpa-*`, restarts via a forced reflow so it replays on apply, and clears everything on `animationName:"none"`.

**Two load-bearing details:** (1) every preset's final keyframe is the neutral resting state (`transform:none; opacity:1`), so `animation-fill-mode:both` never leaves a deck element displaced — this is how preset animations respect the "never overwrite `transform`" invariant (CSS `animation` composes at the computed layer; it never writes the inline `transform`). (2) The keyframe library is injected lazily into the **deck** as `<style id="html-ppt-animations">` on first use — it carries neither the `data-html-ppt-editor` marker nor the editor-style id, so `getCleanHtml()` deliberately **keeps** it and exported decks stay self-contained; it lives in `<head>` so undo (slide-innerHTML only) doesn't drop it.

### AI actions — layout verbs + block library

Two capabilities that give the AI more power while keeping the "never emit HTML" harness (full rationale in `docs/ai-actions-plan.md`). Both flow through the **action envelope** (`shared/editing/actions.ts`): the AI returns `{ message, actions[] }`, `validateActions()` sanitizes per `type` (dropping invalid, never throwing), and `editorApi.applyActions()` applies the whole batch under one undo snapshot. The OpenAI schema models an action as a **flat object** (`type` enum + every field nullable) rather than a discriminated `anyOf`, which is far more robust under strict structured output. `/api/ai-edit` returns this envelope; the bare `{patches}` shape is retired.

- **Layout verbs** (`align/distribute/stack/matchSize/grid/snapToGrid`): the AI emits a *verb + target ids only, never coordinates*. `applyLayoutInternal` reads live `localRect` geometry and applies `{x,y,w,h}` through `applyPatchToElement` — so it inherits `prepareForMove` ("never overwrite transform") and the validator's numeric clamps. `validateLayoutOp` enforces the verb/axis enums, intersects ids with the live selection, and requires per-verb minimum counts. The Toolbar's **Arrange** menu drives the same verbs manually (no AI; ids default to the live selection).
- **Block library** (`shared/editing/blocks.ts`): the AI picks a `blockType` from the vetted 5 (`callout, statCard, bulletItem, quote, labelChip`) and fills **text slots** — it never authors markup. `insertBlockElement` clones the template and fills `[data-slot]` nodes via `setEditableText` (text nodes only). Blocks are self-contained; their base CSS rides the **same persisted-style seam** as animations (`<style id="html-ppt-blocks">`, kept by `getCleanHtml`). The Toolbar's **Block** menu inserts them manually.
- **Scene params** (`shared/editing/scene-params.ts`): tune a deck's **3D / canvas background animation** (the part drawn by deck JS/WebGL that the editor can't touch as CSS). The AI emits a `sceneParam` action — a vetted `key` (`spinSpeed, particleOpacity, keyLightColor, fillLightColor, brightness`) + a clamped number / CSS-safe color — *never code*. A deck **opts in** by exposing `window.__htmlPptScene = { getParams(), setParam(key,value) }`; the editor only ever calls those two methods (`applySceneParamInternal`/`listSceneParams` in `src/editor/scene.ts`). The deck owns applying **and persisting** the change into a kept `<script id="html-ppt-scene">` (no editor marker → survives `getCleanHtml` export, re-applied on load). Decks without the controller report no params, so the Toolbar's **Scene** panel hides its 3D section and `sceneParam` actions no-op. Scene changes live in deck JS, so they're **outside the innerHTML undo stack**. The Tesla sample deck (`sample_deck/Sample_EnglishTeslaPPT/three_scene.js`) is the reference; the deck-author contract is documented in `docs/scene-params-contract.md`.
- **Add a 3D layer to a deck that has none** (`injectThreeScaffold` in `io/project-io.ts` + the `isAdd` branch of `onRegenerateScene` in `App.tsx`): the Scene panel shows an **"✦ Add 3D background"** button when the deck exposes no `__htmlPptScene` controller. It authors a fresh `three_scene.js` via the SAME `/api/regenerate-scene` endpoint (with **no `currentSceneJs`** → the handler writes one from scratch), then `injectThreeScaffold` adds the `<div id="three-canvas-container"><canvas id="three-canvas">`, a kept `<style id="html-ppt-scene-inject">` (behind content, `pointer-events:none`, body transparent + content lifted via `z-index`), the three.js CDN `<script>`, and `<script src="three_scene.js">`. The deck is rebuilt with `buildDeckFromContents` (blob-wires the new file) and the iframe reloads, so `__htmlPptScene` appears and the deck is now AI-editable (sceneParams / scene-regen / per-section scenes) like any 3D deck. AI-deck only (rebuilds + persists from `aiFilesRef`); idempotent (no-op if a `#three-canvas-container` already exists). A deck whose `.slide` has a solid background may need its opacity lowered for the scene to read through — the layer is present and editable regardless.
- **Universal CSS-animation control** (no deck contract): two complementary pieces handle ordinary CSS `@keyframes` motion on *any* deck. (1) The `animationPlayState` patch key (`running`|`paused`) pauses/resumes whatever animation is already on a **selected** element — deck-agnostic, alongside the existing `animationDuration` for retiming. (2) `listBackgroundMotion()`/`applyBackgroundMotion()` in `src/editor/scene.ts` power the Scene panel's **Motion (CSS layers)** group: a global Speed slider + Pause/Resume that scan the animated background layers and write inline `animation-duration` (scaled from the deck's *original* timing, remembered per element so changes never compound) and `animation-play-state`. Inline styles on body-level layers are kept by `getCleanHtml`, so speed/pause survive export (but, like scene params, aren't on the innerHTML undo stack).
- **Chart type** (`shared/editing/chart.ts`, no deck contract): the AI can change a **Chart.js chart's type** by emitting a `chart` action — `id` (the selected chart object) + a vetted `chartType` (`bar, line, pie, doughnut, radar, polarArea`), *never config or code*. `validateChartOp` gates it; `applyChartTypeToElement` (`src/editor/chart.ts`) finds the `<canvas>` on/under the selected element, reads the LIVE instance via the deck's standard `window.Chart` global (`Chart.getChart`), and recreates it with the new type, **preserving the chart's data + options** (it clears per-dataset `type` overrides so a mixed bar+line chart switches cleanly; falls back to data-only options if the preserved options are incompatible with the new type). Deck-agnostic — works on any deck wiring charts through the usual `window.__chartInit`/`new Chart`. **Persistence** rides the same kept-seam pattern as animations/blocks/scene: the chosen type is recorded on the canvas as a `data-chart-type` attribute (not `data-html-ppt-*`, so `cleanHtml` keeps it) and a kept `<script id="html-ppt-chart-overrides">` (no editor marker → kept) is injected once; on the NEXT load it wraps each `window.__chartInit[id]` so after the deck builds the original chart it re-applies the saved type (plus a bounded reconcile for charts already built at load), so export / session-restore reproduce the chosen types. **Caveat** (a chart's type lives in deck JS, not innerHTML/inline style): like scene params it's **outside the innerHTML undo stack** (undo won't revert a type change, and undoing another edit on that slide can blank the canvas — the pre-existing "innerHTML restore drops canvas JS" caveat). The mock engine recognizes type keywords (EN/KO) for keyless demo. Selecting a chart and asking for a data/label change still can't be expressed (the validator drops it → `noValidActionsMessage` explains type-only).
- **Per-section 3D scenes** (additive, optional deck contract): a deck MAY expose `listScenes()` / `getSectionScenes()` / `setSceneForSection(section,name)` alongside `getParams`/`setParam`, backed by a **scene factory + crossfade manager** so the background can differ per section with a smooth ~600ms transition (mapping persisted in the kept `<script id="html-ppt-scene">`). The editor relays via `listSceneSections()`/`applySceneAssignment()` (`src/editor/scene.ts`, no-op when absent → the Toolbar **Scene** panel hides the *"3D scene per section"* dropdowns). Newly generated/regenerated decks get this when the topic has multiple sections (the generation contract in `prompt.ts` + the `/api/regenerate-scene` system prompt both prompt for it); single-scene decks omit the three methods and behave as before. Types: `SceneSectionInfo`/`SceneAssignOp` in `shared/editing/scene-params.ts`; contract documented in `docs/scene-params-contract.md` §3.
- **Scoped element regeneration** (`shared/editing/sanitize.ts`, `api/_editing/element.ts` → `/api/ai-edit-element`): the one **deliberate, controlled exception** to "edits never emit HTML" — the AI rebuilds the **inner HTML of ONE selected element** "from scratch" (advanced restyle/restructure beyond the patch vocabulary). Triple-sanitized: `sanitizeHtml()` (string allowlist) runs server-side (authoritative) AND client-side (`ai/client.ts`), then the editor walks the parsed subtree against `ALLOWED_TAGS`/`ALLOWED_ATTRS` (`scrubSubtree` in `core.ts`). Scoped to one node (the element's tag/classes/`.anim`/geometry are preserved — only children are replaced), applied under one `saveState()` undo snapshot via `editorApi.rebuildElement(id, html)`. Driven by the AiChat **wand** mode (or `REBUILD_INTENT_RE`) → `runAi({elementRegen:true})`. Uses the generation provider layer (Anthropic Opus → OpenAI; mock when keyless).

## Deck generation — "New — AI Deck" (creating, not editing)

A separate flow that **creates** a deck from a topic, vs. the patch/action flow that **edits** an existing one. File ▸ "New — AI Deck" (sparkle icon, `Toolbar.tsx`) opens the wizard (`src/app/components/NewDeckWizard.tsx`: **Topic → Persona → Format → Sample → Details → Review**), POSTs the answers to `/api/generate`, and **auto-loads** the returned deck for micro-editing. Full design rationale: `docs/initial_generation_pipeline.md` (the Korean one-shot master prompt this wires in).

- **Enriched wizard steps** (all optional except Topic): **Persona** — the biggest driver of look: a quick taste picker + a short conversational Q&A (`/api/persona-interview` → `PersonaProfile`, mock = scripted) + reference uploads (kept as `kind:"reference"` images = vision-only style cues, not placed). Persisted reusably in `localStorage` (`slidesmith.persona`). **Format** — `interactive | presentation | document` (`DECK_FORMATS`) + the visual preset. **Sample** — 3 one-slide HTML candidates from `/api/generate-candidates` (parallel `emit_candidate` calls; mock = 3 deterministic variants); the picked one's `{brief, styleCss}` becomes a `candidateSeed` that **locks** the full deck's palette/fonts/preset (`fillBrief` + `seedCssBlock`). **Details** — title + detail + source files. Files (pdf/docx/txt/images) are parsed server-side by `/api/parse-upload` (`api/_generation/parse.ts`, **Node-only**, `mammoth` + `pdfjs-dist`) into **text + extracted images**; extracted/uploaded images become `DeckFiles.assets` (`assets/img-N.ext`) + an image manifest (`collectImageAssets`/`imageManifestText`). **Image placement is app-owned, not model-owned** (`shared/generation/image-slots.ts`): the generation passes never emit `<img>` tags — they only **reserve space + record intent** as an empty slot `<figure data-image-slot data-image-ref="assets/img-N" data-image-intent="…">`. The app fills the real `<img>` (sizing/object-fit/lazy-load/fallback) at load time via `fillImageSlots` in `project-io.ts` — `chooseAssetForSlot` honors the model's `data-image-ref` first, else matches `data-image-intent` against asset captions, else falls back to the next unused asset. Idempotent (skips slots already holding an `<img>`), so it's a no-op on a restored/persisted deck (whose slots are already filled and whose blob refs `normalizeBlobRefs` mapped back to `assets/…`). Runs in both `buildDeckFromContents` (AI decks) and `buildDeckAsync` (disk decks). `project-io.ts` blob-wires `assets` so the iframe resolves them; `storage.ts` writes them to `generated/<id>/assets/`.

- **Trust model:** generating a *new* deck legitimately emits full HTML/CSS/JS — it does **not** go through `validateActions()` (that gate is for edits to a deck the user already trusts). The generated deck renders in the same sandboxed `srcdoc` iframe as any user-opened deck, so it doesn't widen the editor's trust boundary. The "never emit HTML" invariant is about *edits*, not *generation*.
- **Server — multi-pass editorial pipeline** (split under `api/_generation/`: `generate.ts` = orchestration, `providers.ts` = the Anthropic/OpenAI call layer, `assemble.ts` = deterministic build + lint, `mock.ts`, `prompt.ts`, `scene.ts`, `storage.ts`; `api/generate.ts` is the thin route wrapper; dev-mounted in `vite.config.ts`). Holds the API key(s) and runs the FULL `initial_generation_pipeline.md` as four LLM passes the server stitches together: **(1) `emit_plan`** = `DesignBrief` + a slide-by-slide SCQ outline (32–45 slides for "auto"); **(2) `emit_foundation`** = `style.css` + `script.js` + optional `three_scene.js` + a **`componentManifest`** (class names + example markup) documenting exactly what the slide pass may use; **(3) `emit_slides`** = per-section `<section class="slide">` markup + `chartInitJs` snippets, **parallelized by section chunk** (`chunkPlans`, ≤5 slides/chunk); **assemble** (`assembleDeck`, in `api/_generation/assemble.ts`) = the server deterministically builds `index.html` (head/fonts/title, `<main class="presentation">` wrapper, body chrome, CDN-script order — three/chart/datalabels only when used) and stitches every `chartInitJs` onto `script.js` (so Chart.js configs keep their formatter/callback **functions**, unlike JSON); **(4) `emit_qa_fixes`** = the existing lint-gated prompt-only repair. **Mode + robustness ladder:** `HTML_PPT_GEN_MODE` picks the primary path — **default `multi`** (this pipeline; per-call output is bounded, so it can't truncate on a large deck), or `single` = ONE bespoke `emit_deck` call (`generateSinglePass`; most hand-tuned per-slide design but the whole deck must fit in one ≤32K-token response, so it truncates on large decks). **Either mode auto-falls back to the OTHER, then to `mockDeck`** (so a truncated `single` recovers via `multi` instead of degrading straight to the mock template); generation never hard-fails. Check `_log.json` `stage` (`done` / `done-multipass-fallback` = real deck; `mock-fallback` = both failed). **Provider:** `resolveProviders()` (`api/_generation/providers.ts`) prefers **Anthropic Opus** (pinned via `HTML_PPT_ANTHROPIC_MODEL`, default `claude-opus-4-8`) and `callWithFallback()` retries the identical call on **OpenAI** if any Anthropic call fails (empty balance/network) — logged as `*Fallback`/`*Error_*`. `HTML_PPT_GEN_PROVIDER` flips the preference order. Per-pass `max_tokens` via `HTML_PPT_{PLAN,FOUNDATION,SLIDES,QA}_MAX_TOKENS` (OpenAI clamps to ~32K in `callOpenAi`). The big static system prompt (`api/_generation/prompt.ts`) is `cache_control: ephemeral` and is the **same cached prefix for every pass**. The QA pass is auto-gated by `deckLint()` (`HTML_PPT_QA_MODE=auto|always|off`). **No key** → deterministic **mock deck** (`mockDeck`) so the flow works secret-free. **`HTML_PPT_AI_MOCK=1` is now "prompt-export mode"** (free local testing via Claude Code instead of the API, full contract in `docs/ai-initial-generation-pipeline.md`): `handleGenerate` short-circuits before any provider call and returns a `PromptExport` (the exact single-pass generation prompt + `deckId`) instead of a deck — `saveExportRequest` persists `_request.json` + `_generate-prompt.md` + image assets to `generated/<deckId>/`; the user runs the prompt in a Claude Code session that writes the 4 files there; the wizard's **"Load it"** button pulls it back via `/api/load-generated` (`api/_generation/load.ts` → `readDeck`). The AI **edit** path mirrors this: `handleAiEdit` returns an `editExport` JSON (selected element outerHTML + user prompt + `deckId` + `contextFiles`) that the AiChat renders as a copyable block with a **"Reload deck"** action. Intermediate wizard sub-steps (persona/candidates/images) keep their scripted mock so the wizard still flows.
- **Design spec is the full editorial pipeline** (`PIPELINE_DOC` in `api/_generation/prompt.ts` = `docs/initial_generation_pipeline.md` adapted to this single-call env): topic-driven preset (default **light editorial**: warm ivory + gold + serif; dark/tech presets only when the topic fits), SCQ narrative, **per-slide-unique components** (no repeated 2-col cards), 480px watermark divider numbers, per-slide header (Korean section + English italic eyebrow) + footer (page · source), the §4 copy rules (banned words, declarative titles, period rules, easy words, `keep-all`), Chart.js + **chartjs-plugin-datalabels** with **`responsive:true;maintainAspectRatio:false`** (never fixed-px canvases), no section badges. The gold-standard reference is `sample_deck/Sample_KoreanPPT`. 3D is optional/topic-driven (the old always-on plexus is gone).
- **Generated decks follow the editor contract** (`GENERATION_CONTRACT` + the assembler enforce it): `<main class="presentation">` carries the scroll-snap directly (no `slide-container` — that v2 class is retired; `core.ts` already speaks the `.slide.in-view .anim` reveal dialect the editorial decks use); `.slide` 100vh; reveal via `.anim`/`.anim-N` toggled by `.slide.in-view`; never inline-freeze `transform`; charts wired through `window.__chartInit[canvasId]`; the server injects body chrome (`.aura-follower`, `.progress-bar`, `.slide-indicator`) the foundation styles+wires; 3D (when used) wrapped in `#three-canvas-container` exposing `window.__htmlPptScene` (NOT `window.threeScene`; `transitionThreeScene` optional); `@media print` + `@page 1920×1080` + headless `initAll`. `deckLint()` (`api/_generation/assemble.ts`) enforces these on the assembled deck (incl. a fixed-px-canvas check).
- **Storage seam** (`api/_generation/storage.ts`, **Node-only**): writes `generated/<deckId>/{index.html,style.css,script.js,three_scene.js,assets/*,_brief.json,_log.json,_request.json,_plan.json,_prompts.json}` — the stand-in "backend" until a real one exists (swap this one module). `_request.json` is the verbatim user wizard input; `_plan.json` is the multi-pass outline (debug trail). **`_prompts.json` is the durable prompt history** — every generation pass's prompt (`gen.prompts` → `saveDeck`) AND every later **modification** prompt (edit / scene-regen / element-regen), appended via `appendPrompt`/`/api/deck-prompts` (`api/_generation/prompt-log.ts`); the client posts each edit prompt from `App.tsx` (`appendDeckPrompt`, keyed by `lastDeckIdRef`). Browser `localStorage` (`slidesmith.chat.*`) is now just the fast UI mirror. Generation never fails on a write error (logged; contents still returned). `generated/` is gitignored. Because it uses `node:fs`, `/api/generate` (and parse/candidates/persona/deck-prompts) need a **Node** runtime — dev middleware now; prod needs Node functions, not edge.
- **Client:** `requestGeneration()` (`src/app/ai/generate-client.ts`) → `buildDeckFromContents()` (`io/project-io.ts`) makes an in-memory `LoadedDeck` (no `dirHandle`, so Save falls back to download). `downloadDeckFiles()` + the File-menu "Download AI source files" item let a user pull the raw 4 files on demand.
- **Generation memory** (keeps the "session alive" for edits): the AI's compact `DesignBrief` is persisted per deck in `localStorage` (`slidesmith.brief.<deckSignature>`, same keying as chat threads in `App.tsx`) and threaded into every AI edit (`requestAiActions(prompt, ctxs, brief)` → `deck_brief` in the `/api/ai-edit` payload) so the editing model remembers the deck's original palette/fonts/voice. Serverless-friendly + low-token (a brief, not the transcript).
- **Logging** (`src/app/lib/logger.ts`): a capped `localStorage` ring buffer + `window.onerror`/`unhandledrejection` capture + user "Download logs" (wizard error footer). User feedback → `feedback` entries. Server side, each generation writes `_log.json` (token usage, stage, duration, errors) — the maintenance trail.
- **Session restore + welcome screen:** the working deck is auto-persisted (`slidesmith.lastDeck` in `localStorage` — the client-side stand-in for the future per-user backend) on generation and debounced after edits (capturing the live clean HTML via `getCleanHtml`, with blob refs normalized back to relative paths since the editor never touches `style.css`/`script.js`), and **restored on launch** so the user lands in their deck. Only AI decks are persisted (we have their text); opening a disk folder clears it. With no last deck, the empty canvas shows a friendly **"Create with AI"** welcome (`App.tsx` `.deck-empty`), not the old techy "Open Folder" instructions — the AI wizard is the primary path.
- **Web research (PLAN pass):** the plan/outline pass enables the provider's **live web-search tool** (`HTML_PPT_WEB_SEARCH`, default on) so the brief/outline carry real sourced stats + brand colors (every slide footer cites a source). OpenAI = the built-in `web_search` tool on the Responses API (coexists with the json_schema output); Anthropic = the `web_search_20250305` server tool with `tool_choice:auto` (forcing the output tool would skip the search). Search count is logged as `planWebSearches`. Only the plan pass searches (foundation/slides render from its findings) to bound cost/latency.
- **The contract lives in `shared/generation/`** (barrel `@shared/generation`, split into `types.ts` = request/`DeckFiles`/`DesignBrief`/`GeneratedDeck`/`DeckPlan`/`SlidePlan`/`FoundationResult`/`SlideHtml` types + scene-regen types, `schemas.ts` = the multi-pass schemas `EMIT_PLAN_SCHEMA`/`EMIT_FOUNDATION_SCHEMA`/`EMIT_SLIDES_SCHEMA` and `coercePlan`/`coerceFoundation`/`coerceSlides` plus the fallback `EMIT_DECK_SCHEMA`/`QA_SCHEMA` and `EMIT_SCENE_SCHEMA`, `wizard.ts` = the wizard menus + `GenerationRequest`, `pricing.ts` = token/cost accounting). 3D scene **regeneration** (author a brand-new `three_scene.js`) lives in `api/_generation/scene.ts` → `/api/regenerate-scene`. **Render-and-measure QA** (`api/_generation/render-qa.ts`, single mode): `measureDeck()` renders the assembled deck in headless Chromium, reveals every slide, and MEASURES layout failures the string-only `deckLint()` can't see — slides whose content overflows the 100vh frame (vertical/horizontal). Findings are folded into the SAME `emit_qa_fixes` call `generateSinglePass` already makes, so it costs **~0 extra model tokens** (render + measurement are free; only the repair call, which already runs, spends tokens). Playwright is a **soft dependency** — if it (or its browser) isn't installed, `measureDeck` returns `[]` and the pipeline degrades to text-only lint QA, never hard-failing. Activate with `npm i -D playwright && npx playwright install chromium`; toggle/tune via `HTML_PPT_RENDER_QA{,_TIMEOUT_MS,_SETTLE_MS,_OVERFLOW_PX}`. **Still deferred:** the *vision* half (screenshot → model critique) and PDF export, plus a serverless-compatible browser backend (the render loop runs in the Node dev server out of the box; prod needs a chromium-capable runtime).

## Architecture / data flow

```
                         ┌──────────────── React shell (parent document) ────────────────┐
 Toolbar ─┐              │  App.tsx  ──uses──> useEditorBridge(iframeRef)                 │
 Inspector ├─onPatch──>  │             │            │  postMessage Command {id,method,args}│
 AiDialog ─┘             │             │            ▼                                      │
   │ requestAiPatch      │      applyPatch(patch) ──┴──────────────┐                       │
   ▼                     └───────────────────────────────────────┼───────────────────────┘
 POST /api/ai-edit                                                │  (iframe boundary)
   │  (server holds OPENAI_API_KEY, runs validator)               ▼
   └─> safe patch ─> client re-validates ─> applyPatch     ┌─ deck <iframe> (srcdoc, same-origin) ─┐
                                                           │  injected editor bundle:              │
 events back to React  <──postMessage evt {selection|      │   src/editor/* (core, rpc, state)     │
 (selection/mutation/slide/log/ready)                      │   window.__htmlPptEditor = editorApi  │

```

- **`src/editor/`** — the editor itself (a TypeScript port of the original `editor_bridge.js`). Runs inside the iframe. `core.ts` holds the selection/geometry/overlay/handler engine and the `editorApi` object; the self-contained leaf concerns are extracted into one-directional modules `core.ts` imports (they never import core): `dom-utils.ts` (pure CSS/DOM helpers), `text-nodes.ts` (the TreeWalker text-node discipline — `setEditableText`/`getEditableText`), `animation.ts` (`applyAnimation` + the `html-ppt-animations` style seam), `scene.ts` (scene params + background-motion control). `rpc.ts` is the iframe side of the postMessage protocol; `events.ts` decouples emit from transport; `state.ts` holds `STATE` + the selector tables; `history.ts` is the snapshot undo/redo; `index.ts` is the IIFE entry that guards double-injection and sets `window.__htmlPptEditor`.
- **postMessage RPC replaces QWebChannel.** Parent → iframe sends `Command`s (`src/types/messages.ts`); methods that return values (`getCleanHtml`, `getSelectedContext`) resolve via a `Map<id, resolve>` in `useEditorBridge.ts`. iframe → parent sends fire-and-forget `EditorEvent`s (`selection`/`mutation`/`slide`/`log`/`ready`). srcdoc iframes are same-origin, so injection works and origins can be pinned.
- **`src/app/`** — the shell. `App.tsx` (orchestrator, ≈ the old `main_window.py`), `components/` (`DeckFrame` loads the deck + injects the bundle; `Toolbar` is a grouped icon bar; `Inspector` is a **floating panel** rendered only while something is selected — it auto-applies edits debounced 120ms and its × calls `bridge.deselect()`; `AiPromptDialog`; `ShortcutsHelp`), `hooks/` (`useEditorBridge` bridge client, `useKeyboardShortcuts`), `io/project-io.ts` (File System Access API), `ai/client.ts` (calls the proxy + re-validates).
- **`api/`** — the serverless functions. The route files stay at `api/` root (their paths map to URLs); the logic lives in `_`-prefixed folders Vercel ignores for routing, split by domain: **`api/_editing/`** (`handler.ts` = the text-edit endpoint that reads `OPENAI_API_KEY`, calls the OpenAI Responses API with the shared schema, runs the validator, falls back to a deterministic **mock** if no key; `image.ts` = image generation + web image search; `common.ts` = shared `env`/`extractResponseText`/`ContextLike`) and **`api/_generation/`** (see the generation section). `ai-edit.ts`/`ai-image.ts`/`ai-image-search.ts` are the web-standard edge wrappers (importing `_editing/*`). In dev, `vite.config.ts` mounts the same handlers at `/api/ai-edit` etc. so AI works with no separate backend.

## Editor hard constraints (preserve when editing `src/editor/`)

These behaviors are load-bearing — the original deck animations and DOM integrity depend on them:

- **Never reparent elements and never overwrite `transform`.** Decks animate via `.anim` opacity/transform transitions and `.slide.in-view`. `prepareForMove()` only sets `position:relative` (if static) + `left`/`top`/`zIndex`; movement adjusts `left`/`top`, never `transform`.
- **Selection must not mutate the DOM.** A plain click only highlights; `prepareForMove` runs lazily on the first >5px drag.
- **Text editing rewrites text *nodes*, not `innerHTML`.** `setEditableText` (`src/editor/text-nodes.ts`) walks `TreeWalker(SHOW_TEXT)` and maps newline-separated lines onto existing text nodes, preserving nested `<strong>/<span>/<cite>`. `textSafe` in the payload signals whether text editing is offered.
- **Two-tier selection targeting** (`normalizeTarget`): prefer known deck classes in `BLOCK_SELECTOR`, else fall back to `isGenericEditableObject` heuristics; `NEVER_SELECT_SELECTOR` blocks structural chrome (clicking the Three.js canvas selects `#three-canvas-container`).
- **Viewport-based slide tracking** via `IntersectionObserver` + `bestVisibleSlide()`, expecting `.slide` inside `.presentation`; re-entering a slide replays its animation by toggling `.in-view`.
- **Edit Mode OFF is a clean presentation view**: `setEditMode(false)` clears selection, handle, the orange dashed `data-html-ppt-live-edit` outline, contenteditable, and drag/resize state.
- **`getCleanHtml()`** clones the DOM and strips every editor artifact (`[data-html-ppt-editor]`, the injected style, `contenteditable`, `data-html-ppt-*`) before save/export.

## Keyboard shortcuts & undo/redo

PowerPoint-style shortcuts: Ctrl/⌘ Z/Y (undo/redo), C/X/V (copy/cut/paste via an internal clipboard in `STATE.clipboard`), D (duplicate), Delete/Backspace, arrow nudge (Shift = 10px), Ctrl ] / [ (z-order), Esc (deselect/finish text).

**They are handled in two places, by design** — and must stay in sync:
- `src/editor/core.ts` `onKeyDown` handles events when focus is inside the deck iframe.
- `src/app/hooks/useKeyboardShortcuts.ts` handles them when focus is in the React shell, forwarding to `editorApi` via the bridge.
The two never double-fire (separate documents), but adding a shortcut means adding it to *both*. The parent hook ignores events while a form field is focused.

**Undo/redo** (`src/editor/history.ts`) is snapshot-based: `saveState()` pushes the slide container's `innerHTML` before each discrete mutation (drag/resize gesture start, applyPatch, insert, duplicate, delete, z-order, text-edit start, paste; arrow nudges are coalesced). It emits a `history` event (`{canUndo,canRedo}`) that drives the toolbar buttons. Caveat: restoring `innerHTML` re-creates nodes, so deck-attached listeners inside a slide (e.g. a Three.js canvas) are dropped on undo of that slide.

## Slide management (filmstrip panel + insert/reorder/AI slide)

A PowerPoint-style **Slides panel** (`src/app/components/SlidesPanel.tsx`), toggled by a filmstrip icon in the `Toolbar` (`onToggleSlides`/`slidesOpen`), renders a rail beside the deck (`.deck-pane.with-slides` → `.slides-panel` + `.deck-stage`). It's a **lightweight outline**, not pixel thumbnails: each card shows the slide number + a title snippet (`SlideSummary`). Click a card to navigate; hover for per-slide actions (move up/down, duplicate, delete); the **"+" between cards** inserts a slide there — **Blank**, **Duplicate this**, or **✦ AI slide…** (a prompt box).

- **Editor side** (`src/editor/core.ts`): structural ops on the `.presentation` exposed on `editorApi` and the bridge — `listSlides()` (read-only), `insertSlide(InsertSlideSpec)`, `deleteSlide(index)`, `moveSlide(from,to)`. Each mutator requires edit mode, records ONE `saveState()` undo snapshot, re-runs `observeSlide()` (now a single **persistent** `IntersectionObserver` reused across inserts so no observer leak), and re-emits `slide`/`mutation` (→ autosave). `insertSlide` kinds: `blank` (a centered `.slide` with `.anim` placeholders), `duplicate` (clean clone — `cleanSlideMarkers` strips editor/AI-thread attrs), `html` (AI markup, run through `sanitizeHtml` + `scrubSubtree` before it touches the DOM, wrapped in `<section class="slide">` if needed). `deleteSlide` refuses to remove the last slide. Types: `SlideSummary` (`types/context.ts`), `InsertSlideSpec` (`types/messages.ts`).
- **AI single-slide GENERATION** (`api/_generation/slide.ts` → `/api/generate-slide`, `EMIT_SLIDE_SCHEMA`/`coerceSlideResult` in `shared/generation`): authors ONE `<section class="slide">` from a prompt + the deck brief + the neighbour slides' outerHTML (style/component context), so it looks native. **Static only** — no `<script>`/`<canvas>`/Chart.js (the inserter can't wire deck JS; charts/diagrams must be inline SVG/CSS). Same trust model as scene-regen/initial generation (emits HTML, sandboxed iframe) BUT the editor **also sanitizes** it on insert since it lands in a trusted deck. Provider layer reused (Anthropic Opus → OpenAI; deterministic mock when keyless). Client: `requestSlide()` in `ai/generate-client.ts`; `App.tsx` `onAiInsertSlide` fetches neighbour HTML via `getCleanHtml`, inserts via the bridge, logs the prompt (`appendDeckPrompt`, kind `edit`).

## Deck loading & save/export (`io/project-io.ts`)

- **Open Folder** (Chromium): File System Access API; enables *Save HTML* in place and *Export Project*. **Open Files…**: `<input webkitdirectory>` fallback (any browser); save falls back to download.
- The deck is rendered via a same-origin `srcdoc` iframe with assets rewritten to blob URLs. **Known limitation:** only `src`/`href` attributes are rewritten — CSS `url(...)` references inside stylesheets are not yet, so relative font/image refs in a deck's CSS may not resolve in preview.
- *Save HTML*, *Export HTML Only*, and *Export Project* all read the live DOM via the bridge's `getCleanHtml()`.

## Conventions

- Korean is a first-class UI language for AI prompts; the mock-mode keyword matching in `api/_editing/handler.ts` is bilingual (고급/premium, 크게/bigger, 가운데/center). Don't strip the Korean keywords.
- `validatePatch` is the trust boundary for untrusted (AI) data — keep new patch handling routed through it, and prefer dropping bad values to throwing.
- The editor only ever writes inline `style="..."` (and text nodes) onto deck elements; it never edits the deck's CSS/JS files.
- `sample_deck/` is a sample HTML deck kept as test content — load it via Open Folder.
